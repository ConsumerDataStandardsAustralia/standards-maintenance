<p>These principles articulate qualitative outcomes that the API definitions should seek to deliver.</p>
<p>The API definitions will consider and incorporate the need for a high degree of security to protect customer data.  This includes the risk of technical breach but also additional concerns of inadvertent data leakage through overly broad data payloads and scopes.  The security of customer data is a first order outcome that the API standards must seek to deliver.</p>
<p>To ensure that the entry hurdle for new developers is low the experience of the developers that are building clients using the APIs will be considered.  The ability for a developer to easily understand and write code using the APIs in modern development environments should be facilitated by the API standards.</p>
<p>These principles articulate specific technical outcomes that the API definitions should seek to deliver.</p>
<p>The underlying implementation of the APIs should not be constrained or driven by the API definitions and standards.  Conversely, the underlying implementation choices should not be visible or derivable to the client applications using the APIs.</p>
<p>As complexity will increase implementation costs for both holders and clients as well as reduce the utility of the APIs, API definitions should seek to be as simple as possible but no simpler.</p>
<p>As the APIs are defined care should be taken to ensure that the data payloads defined represent rich data sets that can be used in many scenarios, including scenarios not necessarily front of mind during the design process.</p>
<p>The API definitions should consider and incorporate performance implications during design ensuring that repeated calls are not necessary for simple use cases and that payload sizes do not introduce performance issues.</p>
<p>The API definitions across the full suite of APIs should be consistent with each other as much as possible.  Where possible common data structures and patterns should be defined and reused.</p>
<p>The API definitions and standards should be built for extensibility. This extensibility should accommodate future API categories and industry sectors but it should also allow for extension by data holders to create unique, value add offerings to the ecosystem.</p>

<li><strong>Data Recipient</strong>: For the purposes of these NFRs a data recipient is defined as a configured application presented in the register meta data.  This acknowledges that a single accredited entity may be able to register multiple independent services (or apps) that can obtain authorisations from consumers independently of each other.</li>
<li><strong>Unauthenticated</strong>: API requests to API end points that the standards deem to be publically available.  This implies that these end points may be accessed by any client without the client performing any authentication or authorisation actions</li>
<p>A expiry time of a unique session should be set according to the statements included in the <a href="#security-profile">Security Profile</a>.</p>
<p>After a unique session is expired it is expected that the data recipient, for the same customer, may establish a new session as long as the authorisation is still valid.</p>
<p>The availability requirement does not include planned outages.  Planned outages should be:</p>
<li>May occur without notification if the change is to resolve a critical service or security issue.</li>
<p>Note that calls initiated in excess of a traffic threshold (see next section) may be excluded from the performance requirement.</p>
<li>Services should be designed to minimise traffic with data holders</li>
<li>Services should be designed to be resilient in the case of the rejection of a call by a data holder due to traffic threshold breaches</li>
<li>Services should schedule unattended calls to avoid high traffic periods</li>
<li>Unattended calls should be managed to avoid short term bursts of traffic</li>
<p>Within this proposal there is no specific requirement with regard to data latency (ie. how up to date data should be).  Instead, the requirement for data latency is that data presented via API end points should be commensurate to data presented via other primary digital channels.</p>
<p>For example, for a Bank that provides a mobile application as their primary digital experience, a balance presented via one of the balance end points should be the same as the balance presented through the mobile application.</p>
<li>Periods of time when the digital channels for the data holder are the target for a distributed denial of service or equivalent form of attack (this should result in http error <code>429 Too Many Requests</code> being returned).</li>
<li>A significant increase in traffic from a poorly designed or misbehaving data recipient (this should result in http error <code>429 Too Many Requests</code> being returned).</li>
<li>If the data holder identifies a situation where there is the potential for physical or financial harm or abuse (this should result in http error <code>403 Forbidden</code> being returned).</li>
